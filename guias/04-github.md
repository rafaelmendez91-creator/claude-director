# Guía 4 — GitHub explicado para un director

## Qué es, en una frase

GitHub es un **archivador con historial completo**: cada documento guarda todas
sus versiones, con fecha, autor y el porqué de cada cambio. Nada se pierde,
nada se pisa, y siempre se puede volver a como estaba antes.

## Por qué le conviene a usted (aunque no escriba una línea de código)

- **Fin del "versión final FINAL (2).xlsx"**: hay UNA copia viva de cada cosa,
  y el historial guarda las anteriores.
- **Trazabilidad**: "¿quién cambió esta cifra y cuándo?" tiene respuesta exacta.
- **Trabajo en paralelo sin pisarse**: dos personas (o dos asistentes) trabajan
  cada uno en su copia y los cambios se integran de forma controlada — es
  exactamente como el equipo de Rafael coordina varias sesiones de Claude a la
  vez sin que se mezclen.
- **Su asistente lo usa como memoria firme**: al cierre de cada sesión guarda
  una "foto" del estado. Un corte de luz no borra nada.

## Cómo lo usa usted (sin aprender comandos)

Usted solo habla; su asistente ejecuta:

- *"Guarda una versión de esto"* → guarda la foto con una descripción legible.
- *"Muéstrame cómo estaba este documento el mes pasado"* → se la trae.
- *"¿Qué cambió esta semana en la carpeta del proyecto?"* → resumen en una página.
- *"Clónate la carpeta que me pasó Rafael"* → la descarga y queda sincronizada.

## Las 3 palabras que oirá (traducidas)

- **Repositorio**: la carpeta del proyecto con su historial incluido.
- **Commit**: una foto guardada del estado actual, con fecha y nota.
- **Clonar**: descargar una copia completa de un repositorio a su computadora.

Con esas tres le alcanza. Todo lo demás es mecánica de su asistente.
